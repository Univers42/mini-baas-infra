### 1.1 Authentication Routes (via GoTrue)

| Endpoint | Method | Protected | Purpose |
|----------|--------|-----------|---------|
| `/auth/v1/signup` | POST | API Key | User registration |
| `/auth/v1/token?grant_type=password` | POST | API Key | Login (access + refresh tokens) |

| `/auth/v1/health` | GET | API Key | Auth service health check |

**Request Headers (All):**
- `apikey: <ANON_KEY>` (gateway validation)
- `Content-Type: application/json`

**Response:**
- Success (200/201): `{ user, session_token, access_token, refresh_token, expires_in }`
- Error (4xx): Standard error envelope

**Request Headers (All):**
- `apikey: <ANON_KEY>`
- `Authorization: Bearer <JWT>`

### 1.3 MongoDB Data Routes (Custom HTTP Service)

**Base Path:** `/mongo/v1`

| Endpoint | Method | Auth | Purpose |
|----------|--------|------|---------|
| `/mongo/v1/health` | GET | API Key | MongoDB service health |
| `/mongo/v1/collections/:name/documents` | POST | JWT | Create document (owner_id auto-injected) |
| `/mongo/v1/collections/:name/documents` | GET | JWT | List documents (RLS via owner_id) |
| `/mongo/v1/collections/:name/documents/:id` | GET | JWT | Get single document |
| `/mongo/v1/collections/:name/documents/:id` | PATCH | JWT | Update document |
| `/mongo/v1/collections/:name/documents/:id` | DELETE | JWT | Delete document |

**Request Headers (All):**
- `apikey: <ANON_KEY>`
- `Authorization: Bearer <JWT>` (protected routes)

**Response Envelope (Mongo):**
```json
{
  "success": true,
  "data": { /* payload */ },
  "error": null,
  "meta": { "request_id": "uuid", "pagination": { "limit": 20, "offset": 0, "total": 100 } }
}
```

**Validation Rules:**
- Collection name regex: `^[a-zA-Z0-9_-]{1,64}$`
- Max payload: 256 KB
- Forbidden keys: `_id`, `owner_id` (server-controlled)

## 2. Demo Data Models

### 2.1 PostgreSQL: Projects Table

**Schema (Current in db-bootstrap.sql):**

```sql
CREATE TABLE public.projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  status TEXT DEFAULT 'active' CHECK (status IN ('active', 'paused', 'archived')),
  owner_id TEXT NOT NULL,  -- JWT subject
  description TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

ALTER TABLE public.projects ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can CRUD their own projects"
  ON public.projects
  FOR ALL
  USING (auth.uid()::text = owner_id)
  WITH CHECK (auth.uid()::text = owner_id);
```

**Demo Data (Creation via Signup Flow):**

```json
{
  "name": "Mobile App MVP",
  "status": "active",
  "description": "Build authentication prototype",
  "owner_id": "user-uuid-from-jwt-subject"
}
```

**Expandable Tables (Post-MVP):**
- `public.user_profiles` (extended user info)
- `public.posts` (content model example)

**User Profiles (optional extension):**

```sql
CREATE TABLE public.user_profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  bio TEXT,
  avatar_url TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
```

### 2.3 MongoDB: Document Schema

**Database:** `mini_baas`  
**Collection:** User-defined (e.g., `tasks`, `notes`, `events`)

**Document Structure Example (tasks):**

```json
{
  "_id": "ObjectId (generated by MongoDB)",
  "owner_id": "user-uuid-from-jwt (injected by service)",
  "title": "Complete API docs",
  "description": "Write endpoint documentation",
  "status": "in_progress",
  "priority": "high",
  "due_date": "2026-04-05T23:59:59Z",
  "tags": ["docs", "urgent"],
  "created_at": "2026-03-31T10:00:00Z",
  "updated_at": "2026-03-31T15:30:00Z"
}
```

**Constraints:**
- `owner_id` is **server-injected** from JWT subject; client cannot override
- No `_id` in create request; MongoDB generates it
- All query/list operations filter by `owner_id` automatically

**Collection Examples (MVP Demo):**

| Collection | Use Case | Sample Fields |
|-----------|----------|----------------|
| `tasks` | Project task tracking | title, status, priority, due_date, owner_id |
| `notes` | User notes/docs | content, tags, is_pinned, owner_id |
| `events` | Event logs | event_type, timestamp, metadata, owner_id |

## 3. Schema Contract Variations

### 3.1 Acceptable Variations

#### PostgreSQL Tables
- **Extension 1:** Add columns to `projects` (e.g., `budget`, `client_name`) → backward compatible
- **Extension 2:** Create new tables with same RLS pattern → independent from projects
- **Constraint Change:** Tighten enum on `status` → requires migration, inform team
- **Not Allowed (MVP):** Cross-table joins, aggregations, or temporal queries → defer to phase 2

#### MongoDB Collections
- **Extension 1:** Different document shapes per collection → allowed (schemaless)
- **Extension 2:** Add nested objects/arrays → allowed
- **Constraint:** `owner_id` must always be top-level flat field (not nested) → enforce in service
- **Not Allowed (MVP):** Aggregation pipelines, transactions → defer to phase 2

### 3.2 Schema Versioning

**Approach:** Immutable document schema with backward compatibility.

**If schema changes are needed:**

1. **Postgres:** Apply schema migration to `db-bootstrap.sql` → recreate volume for fresh deploy
2. **MongoDB:** New collections can use new schema; old collections read as-is (schemaless)
3. **API Contract:** Document breaking changes in [services/contracts/](services/contracts/) READMEs

### 3.3 Cross-Engine Differences

| Aspect | PostgreSQL | MongoDB |
|--------|------------|---------|
| **Schema Enforcement** | Strict (table + column types) | Flexible (document shape) |
| **Ownership Check** | RLS policy at DB layer | Service-layer logic |
| **Default Values** | SQL defaults | Application layer |
| **Pagination** | Offset/limit via PostgREST query | Limit/offset in service |
